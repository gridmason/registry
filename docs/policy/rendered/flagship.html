Registry policy — Gridmason Registry (flagship)

Flagship instance — invite-only launch. Registry endpoint: https://registry.gridmason.dev. Operated by Sniper7Kills LLC, stewards of the Gridmason project.

This page states the policy under which this Gridmason Registry instance reviews, signs, and serves widget and plugin remotes. The review mechanism is part of the open-source platform and is identical on every instance; the policy — what this operator accepts, how it staffs review, and how it handles revocation — belongs to this operator and is published here in full. There are no secret rules (SPEC §4).

A registry is supply chain, never a control plane (SPEC §1). Hosts reach it over outbound 443 (or signed offline bundles) and keep working when it is unreachable; nothing here lets this registry reach into a running host.

1. What this registry serves

Signed, content-hashed ES-module remotes served immutably from a CDN. The reviewed hash is the runnable artifact — a publisher cannot swap code after review. Every served version carries a dual signature and a public transparency-log entry (see §2).

2. Trust model

Every published version carries two signatures, verified by hosts together with the content hash and transparency-log inclusion before load:

  • Publisher signature — authorship, bound to the publisher's OIDC identity at publish time.
  • Registry countersignature — this operator's approval, applied only after review passes. The countersign key is held separately from review staff.

Transparency log: This instance anchors to the public Sigstore infrastructure (Fulcio for keyless certs, Rekor as the log) rather than operating its own CA/log (SPEC §2, GW-D17). Anyone can independently verify what this registry has shipped.

3. Review policy

The flagship's review policy is published in full here. It opens invite-only: only verified publishers may publish while the review roster is single-person (see the launch-phase disclosure in §4).

Automated checks run first on gridmason publish (malicious-code and SDK-adherence analysis, manifest lint, capability diff, dependency-DAG check). These are the same checks gridmason lint runs locally, so a publisher sees the automated verdict before submitting. Human review then applies by trust tier:

T1 — declarativeNo executable content — layouts, page types, dashboards.2 days

Capability-increase re-review: 3 days (target). Expedited security-patch lane: 24 hours (target). Appeals go to a second reviewer, never the original (SPEC §4.4).

* SLAs on this instance are published targets, not guarantees — see the launch-phase disclosure in §4. Current measured review latency: published on this page and refreshed as the roster reports; until automated latency reporting lands, the operator updates this figure manually.

4. Separation of duties

Widgets published by this operator's own team ride the same pipeline as everyone else's: reviewed by a different person than the author and countersigned by the separately-held registry key. Operators do not self-approve (SPEC §2).

Launch-phase disclosure — single-roster separation-of-duties waiver (SPEC §4a)

This instance opens invite-only (verified publishers only) with a single-person review roster (1 reviewer). While the roster is below two reviewers, and disclosed here rather than hidden:

  • The §2 separation-of-duties rule is waived for this operator's own published widgets until the roster reaches ≥ 2 reviewers. Every release affected by the waiver is flagged in its transparency-log entry, so the waiver is auditable per release — not just stated here.
  • The review SLAs in §3 are published targets, not guarantees, until review is staffed to ≥ 2 people; the current measured latency is shown in §3.
  • Countersign-key custody: the countersign key is an offline key, held distinct from the reviewer's publishing identity, and its custody procedure is documented before launch — even while both the review and countersign roles are held by one person during the single-roster phase.

Open community publishing and full separation of duties arrive when the roster reaches ≥ 2 reviewers. Self-hosted instances inherit none of this — each publishes its own policy page in this same format.

Review roster & staffing

The flagship launches with a single-person review roster (GW-D18). Its size and the resulting waiver are disclosed above and flagged per affected release in the transparency log. The roster grows to ≥ 2 reviewers before open community publishing (Phase C).

5. Revocation & kill switch

This registry owns distribution state (published / revoked / killed) and publishes it as a signed revocation & kill feed; hosts consume it and fail closed for revocation within the cache TTL (max 24 h). A killed remote never enters a host's import map regardless of local enablement (SPEC §6).

Severity policy: actively-exploited or credential-path issues are killed immediately, with deployment notification within 24 hours; other revocations propagate within 1 hour online (next bundle sync for air-gapped hosts).

6. Contact

Reach the Gridmason project maintainers through the gridmason GitHub organization. Publishing on the flagship is invite-only during the launch phase.

Suspected vulnerabilities in this registry or in a served artifact must be reported privately — never as a public issue. See the security policy.
